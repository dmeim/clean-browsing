# Clean Browsing v1.5.1 — Single-target Ping Monitor

**Release type:** Patch
**Extension version:** 1.5.1
**Target:** Firefox (Manifest V2, gecko min 109.0)

## Overview

v1.5.1 reshapes the Ping Monitor widget. Instead of one widget hosting an
array of targets, **each Ping Monitor widget now represents a single
endpoint**. Place one widget per host you want to watch and arrange them on
the grid however you like.

The visual that used to be the per-target card inside the multi-target
widget is now the entire widget body, plus the 1h / 24h uptime percentages
that used to sit in the multi-target footer.

## What's new

### Single-target widget body

- Status dot, current latency, and a 20-sample response-time sparkline
  scaled against the per-target slow threshold.
- Target label below the graph — there is no longer a separate "Ping
  Monitor" header; the label _is_ the title.
- Inline rolling-uptime row sits below the label, replacing the old
  `HTTPS GET` method tag. The HTTP method indicator stays accessible from
  the Target tab in settings.
- A small **refresh icon** fades in the top-right corner on hover (the same
  pattern as Notes' "Clear completed" action). It hides during edit mode so
  it doesn't compete with the resize/drag affordances.

### Fluid body scaling

The widget body is split into three logical regions — status row, graph,
label + uptime — that all size themselves with CSS `cqmin` against the
widget body as a query container. The result:

- Latency text and target label use the same scaling formula, so they
  grow at the same rate.
- The graph fills its share of the height edge-to-edge instead of sitting
  as a thin line above the label.
- A 1×1 dense-grid cell still shows everything legibly; a large widget
  shows a hero ms number, a tall sparkline, and proportional title /
  uptime text.
- Descenders in the label (the bottom of `g`, `p`, `y`) no longer clip
  inside the ellipsis box at small sizes — `flex-shrink: 0` plus a
  more generous `line-height` keep the line box at full height even
  when the section gets tight.

### Settings

- Three tabs: **Appearance**, **General**, **Target** (renamed from
  `Targets` and stripped of the add/remove machinery).
- **Target** tab is the same form as before — label, address, method,
  check interval, timeout, slow threshold — minus the multi-target list
  scaffolding.
- **General** tab: a new **Uptime windows** picker, the notifications
  toggle, and a **Clear history** action that wipes only this widget's
  history.

### Configurable uptime windows

The footer no longer hard-codes `1h · 24h`. Pick **any two** of `1h`, `4h`,
`8h`, `12h`, `18h`, `24h` from the General tab; both selected windows are
rendered under the graph in ascending order. Tapping an inactive pill
swaps it in for the older of the two active ones — exactly two stay on at
all times. The default pair is still `[1h, 24h]`, and existing widgets
(no stored value) fall back to that pair on first render.

### Default size

The widget defaults to **2×2** in normal-mode cells (4×4 in dense-grid
mode — same physical size) and can't be resized below that. The body
layout doesn't read well at smaller sizes.

This release also makes the grid store treat every widget's
`defaultSize`/`minSize`/`maxSize` as normal-mode cell units and scale them
×2 in dense mode automatically. Adding any widget while dense mode is
on now produces the same physical size as adding it in normal mode, and
`minSize` constraints stay tight in either mode. (Existing instances are
unaffected — `setDenseGrid()` already doubles them when the mode toggles.)

### Documentation

- [`docs/widgets/ping-monitor.md`](../widgets/ping-monitor.md) gains a new
  **"How 'ping' actually works"** section near the top. It spells out that
  the widget is an HTTP `fetch()` (not ICMP), lists the kinds of targets
  that work and don't, and calls out the common gotchas — most notably
  that Tailscale's `100.100.100.100` MagicDNS resolver doesn't serve HTTP
  and therefore can't be monitored this way.

## Breaking changes

**Existing v1.5.0 multi-target ping-monitor instances are removed from the
grid on first load.** All other widgets are untouched. After upgrading,
re-add a Ping Monitor widget per endpoint you want to track and configure
each from its Target tab. Their previous response-time history is not
preserved.

## For contributors

- `PingMonitorSettings` is now `{ target: PingTarget; uptimeWindows: number[]; ... }` instead of
  `{ targets: PingTarget[]; ... }`.
- `PingTargetCard.svelte` is gone; `PingMonitor.svelte` renders the body
  inline and uses CSS container queries (`container-type: size` on
  `.ping-inner`, `cqmin`-based clamps everywhere) to scale fluidly with
  the widget's box.
- `tabs/TargetsTab.svelte` is renamed to `tabs/TargetTab.svelte`.
- `normalizeTarget(t)` no longer takes a legacy settings blob — the
  pre-v1.5.1 multi-target shape is dropped at layout-load time, not
  hydrated. New helper `normalizeUptimeWindows(raw)` validates the
  uptime-windows array on read.
- The migration filter lives in
  [`src/lib/grid/store.svelte.ts`](../../src/lib/grid/store.svelte.ts) as
  `migrateLayout(saved)`, applied inside `load()` before assigning state.
- Per-instance ping behaviour: each widget owns its own interval timer and
  history; multiple widgets pointing at the same URL run independently.
- The grid store now treats `WidgetDefinition.{default,min,max}Size` as
  normal-mode cell units and applies a ×2 multiplier in dense mode. See
  the `denseScale()` helper in `store.svelte.ts`.

## Permissions

No changes from v1.5.0.

## Installation

### From a GitHub release artifact

1. Download `clean_browsing-1.5.1.zip` (or the signed `.xpi`) from the
   [v1.5.1 release page](https://github.com/dmeim/clean-browsing/releases/tag/v1.5.1).
2. In Firefox, open `about:debugging#/runtime/this-firefox`.
3. Click **Load Temporary Add-on…** and pick the `manifest.json` inside
   the extracted zip (or the `.xpi` file directly).
4. Open a new tab to start using Clean Browsing.

### From source

```bash
git clone https://github.com/dmeim/clean-browsing.git
cd clean-browsing
git checkout v1.5.1
npm install
npm run build
```

Then load `dist/manifest.json` via `about:debugging`.
